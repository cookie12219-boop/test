# Reel Atlas — CS551P Advanced Programming, Assessment 3 Part I

A database-driven Flask web application that catalogues ~5,000 films with
linked tables for movies, directors, and genres. Built for the
**CS551P Advanced Programming** module at the University of Aberdeen.

**Live deployment:** https://reel-atlas.onrender.com
(see [Deployment](#deployment) below).

---

## What it does

- Browse a paginated index of films with search and six sort orders.
- Drill into a film’s detail page to see budget, revenue, ROI, cast,
  keywords, production companies, and the film’s percentile rank in the
  full catalogue.
- Place two films **side-by-side** for direct comparison.
- Browse by **genre**: how many films belong to each, the average rating,
  and the genre’s top entries.
- Inspect any director’s filmography via the detail page.

No JavaScript runs in the browser. All interactivity is form posts and
server-rendered Jinja2 templates (per the brief).

### Schema (linked tables)

```
┌─────────────┐  1     N  ┌───────────┐  N     N  ┌─────────┐
│  Director   │◄──────────│   Movie   │──────────►│  Genre  │
└─────────────┘           └───────────┘           └─────────┘
                            │       │
                            └──┬────┘
                          movie_genres (assoc table)
```

A `Director` owns many `Movie`s (one-to-many foreign key). A `Movie`
belongs to many `Genre`s through a proper association table called
`movie_genres`. Genres are queryable through SQL joins, not parsed
strings.

## Running locally

### 1. Set up a virtualenv

```bash
python3 -m venv .venv
source .venv/bin/activate         # macOS / Linux
# .venv\Scripts\activate          # Windows PowerShell
pip install -r requirements.txt
```

### 2. Build the dataset

```bash
python scripts/build_dataset.py
```

This script first tries to download the **TMDB 5000 Movie Dataset** CSV
(a public dataset derived from TMDB’s public API). If that mirror is
unreachable — corporate firewalls, sandboxed CI, no internet — it falls
back to a deterministic 5,000-row synthetic dataset that follows the
same schema. The application is identical either way.

### 3. Run the app

```bash
export FLASK_APP=wsgi.py
export FLASK_CONFIG=development
flask run --debug
```

The development config auto-loads the CSV into a fresh SQLite database
on first request. Visit <http://127.0.0.1:5000/>.

To reload the database manually:

```bash
flask load-data                # load app/config.py:DATASET_CSV
flask load-data --limit 500    # quick subset for testing
```

## Testing

```bash
pytest -v
```

Tests run against an in-memory SQLite database seeded with a tiny
deterministic fixture. They cover the model relationships, the route
handlers (200s and search behaviour), the CSV loader (including malformed

## Deployment

### Render (recommended, free)

1. Push this repo to GitHub.
2. On <https://dashboard.render.com>, choose **New → Blueprint** and
   point it at the repo. Render reads `render.yaml`.
3. Render runs `pip install -r requirements.txt && python
   scripts/build_dataset.py`, then `gunicorn wsgi:app`.
4. The app boots, `create_app()` calls `db.create_all()`, sees an empty
   DB, and auto-loads `data/tmdb-movies.csv` because
   `AUTO_LOAD_DATA=True` in production config.
5. `/healthz` is the health-check endpoint Render polls.

The default free-tier instance keeps SQLite in the instance folder; it
resets on redeploy. For persistence, attach a Render Disk at
`/opt/render/project/src/instance` and set `DATABASE_URL` to a path on
the disk.

### Any WSGI host

```bash
pip install -r requirements.txt
python scripts/build_dataset.py
FLASK_CONFIG=production gunicorn wsgi:app --bind 0.0.0.0:8000
```

## Maintenance

| Task | Command |
|---|---|
| Refresh dataset from CSV | `flask load-data` |
| Drop & recreate tables | `flask init-db` (after wiping `instance/`) |
| Run tests | `pytest` |
| Add a new genre or director | Edit the CSV and re-run `flask load-data` |
| Switch from synthetic to real TMDB | Replace `data/tmdb-movies.csv` with a real TMDB 5000 dump that has the same columns; no code change needed |

### Configuration knobs

| Env var | Default | Purpose |
|---|---|---|
| `FLASK_CONFIG` | `development` | `development` / `testing` / `production` |
| `DATABASE_URL` | `sqlite:///instance/movies.sqlite` | SQLAlchemy URL |
| `DATASET_CSV` | `data/tmdb-movies.csv` | Path to source CSV |
| `SECRET_KEY` | `dev-secret-change-me` | Flask secret; auto-generated on Render |

## Data source

This project follows the schema of the **TMDB 5000 Movie Dataset**
(`tmdb-movies.csv`), itself derived from TMDB’s public API. The fields
match the upstream dataset, so any real TMDB CSV dump drops in with no
code changes. See `app/loader.py` for the column mapping.

The synthetic fallback in `scripts/build_dataset.py` exists purely so
the application can be developed in sandboxed environments where the
mirror is unreachable. It generates 5,000 rows with plausible
distributions of budget, revenue, runtime, and rating.
